# Changelog — 2026-05-27

Lapisan logger di-overhaul end-to-end + konsolidasi env. Semua perubahan additive,
tidak ada breaking change di domain bisnis (router/feature/handler tetap utuh).

> Total: 9 file baru, 13 file diubah, 4 file dihapus.
> Test suite: 42 file, 154 test pass (kecuali `packages/db/**` yang preexisting
> blocked by `better-sqlite3` NODE_MODULE_VERSION mismatch — `npm rebuild`).

---

## Ringkasan tujuan

User minta logger:

1. Real-time, status sesuai severity, tidak silent saat crash.
2. Format terminal mudah dibaca (kolom rapi, warna).
3. Urutan field: timestamp · status · platform · type (PC/GC) · nama+ID · command · msg/err.
4. Mask ID PII di terminal, JSON file tetap raw lengkap.
5. Rotate harian, simpan 7 hari, hari ke-8 yang terlama otomatis dihapus.
6. Timezone via env (Asia/Jakarta default), validated, fail-fast saat typo.
7. Cuma `.env` + `.env.example` di repo (drop variant `.development`/`.production`).

Semua sudah jalan. Detail di bawah.

## Phase 1 — Hardening logger inti

**Masalah awal** (`packages/utils/src/logger.ts`):

- `logDir` default `/home/container/data/log` (Pterodactyl-only) → di Windows dev,
  catch path `apps/*/index.ts` panggil `createRootLogger({ env })` tanpa `logDir`,
  worker `pino-roll` crash silent.
- Tidak ada error handler di `pino.transport({ targets })` → kalau worker mati
  (disk penuh, dll), main thread tidak tahu, log silently hilang.
- `redact.paths` cuma 4 entry (`*.password`, `*.token`, `*.authorization`,
  `*.cookie`, plus 8 path eksplisit env/config). Tidak cover Baileys auth blob,
  apiKey generic, header HTTP.

### Patch 1 — `logDir` default cwd-relative

```diff
- const logDir = options.logDir ?? '/home/container/data/log';
+ const logDir = options.logDir ?? join(process.cwd(), 'data', 'log');
```

Production tetap aman karena `bootstrap.ts` selalu lewat `config.LOG_DIR` dari
env. Yang ke-fix cuma fallback untuk dev/test.

### Patch 2 — Worker error listener

```ts
const transport = pino.transport({ targets });
transport.on('error', (err) => {
  process.stderr.write(JSON.stringify({
    time: Date.now(),
    level: 60,
    status: 'fatal',
    msg: 'pino transport worker error; logs will only reach stderr until restart',
    err: err instanceof Error
      ? { message: err.message, stack: err.stack }
      : { message: String(err) },
  }) + '\n');
});
```

Worker death sekarang produce 1 baris JSON `level: 60` ke stderr — visible via
`docker logs`, `journalctl`, atau panel Pterodactyl error log.

### Patch 3 — Redact extended (24 paths total)

Dulu 4 generic + 8 explicit. Sekarang 24 path mencakup:

```

*.apiKey, *.api_key, *.secret, *.bearer,
*.accessToken, *.refreshToken, *.access_token, *.refresh_token,
*.privateKey, *.private_key, *.creds, *.credentials,
*.signedIdentityKey, *.noiseKey, *.encryptedBlob, *.iv, *.authTag,
headers.authorization, headers.cookie,

config.AUTH_ENCRYPTION_KEY, config.TELEGRAM_BOT_TOKEN, config.OWNER_WA, config.OWNER_TG
```

Mode `remove: true` → field hilang total dari output. Cover Baileys auth shape

**Catatan**: 1 patch `sync: false` di `pino-roll` opts sempat bikin

## Phase 2 — Custom prettifier (Style A)

**Masalah**: format `pino-pretty` default + `messageFormat` template:

```
[2026-05-27 09:14:02.005 +0700] WARN: [wa] owner/eval rejected → Owner only
    eventId: "01KS7GDS0XN5K8M2P3Q4R7Z2A1"
    traceId: "01KS7GDR9XN5K8M2P3Q4R7Z2A1"
    err: { type: "OwnerOnlyError", message: "Owner only", stack: "..." }
```

6 baris per record, eventId 26 char visual sampah, dobel field
(messageFormat sudah print `feature` + `status`, multiline print lagi).

**Solusi**: drop `pino-pretty` total, bikin custom prettifier yang dipasang
sebagai stream langsung di `pino.multistream()`.

### File baru — `packages/utils/src/log-pretty.ts`

API publik:

| Export | Tipe | Fungsi |
|---|---|---|
| `RenderOpts` | type | `{ logPii, color, tz? }` |
| `formatTime(epochMs, tz?)` | fn | `'HH:mm:ss.SSS'` di TZ tertentu (fallback UTC kalau tz invalid) |
| `maskId(id, platform, logPii)` | fn | Mask WA JID `6289****8933`, group `grp:1203****`, Telegram numeric `-100****890`; `@username` kept; raw kalau `logPii=true` |
| `renderLine(log, opts)` | fn | Render 1 record JSON → 1 string compact (multi-line cuma untuk stack) |
| `createPrettyStream(opts)` | fn | `Transform` stream yang parse NDJSON dari pino → renderLine, pipe ke `process.stdout` |

Layout kolom (lebar fixed):

```
HH:MM:ss.l   STATUS   PLAT  TYPE   NAME › ID                          ACTION              │ MSG/ERR
12 char      5 char   2     2      32 char (truncate +…)               18 char              │ flex
                                                                                              ╰─ stack indent
```

Konvensi visual:

- `OK` hijau bold · `DBG` abu · `WARN` kuning bold · `ERROR` merah bold ·
  `FATAL` merah bold + bg merah.
- Platform `wa` cyan · `tg` cyan · system `—`.
- Type `PC` magenta · `GC` magenta · system `—`.
- Command `/xxx` cyan bold.
- Stack `╰─` dim.
- Separator `│` dim.

Status mapping (level number → status string fallback kalau `status` field tidak
ada):

```
10 trace → ok    20 debug → ok    30 info → ok
40 warn  → rejected    50 error → error    60 fatal → fatal
```

Mixin merge strategy lama dipertahankan: kalau call site set
`status: 'rejected'` eksplisit, override default `'ok'`.

### Fix `err.type === 'Object'`

Pino default error serializer ambil `error.constructor.name`. Untuk plain object
`{ type: 'XError', message: '...' }` itu jadi `'Object'`, tampil nyampah di
pretty: `Object: ENOSPC: ...`. Patch:

```ts
function errMessage(err: unknown): string {
  if (!err || typeof err !== 'object') return '';
  const rawType = pickString((err as { type?: unknown }).type);
  const type = rawType && rawType !== 'Object' ? rawType : '';
  ...
}
```

`'Object'` di-skip, output rapi: `ENOSPC: no space left on device`.

### Wiring di `logger.ts`

Logger function jadi **async** (breaking ke call site):

```ts
export async function createRootLogger(options: RootLoggerOptions = {}): Promise<Logger>
```

Alasan: `pino-roll` v4 returnnya `Promise<Writable>`. Awalnya pakai
`pino.transport({ targets: [pino-roll] })` di worker thread, tapi
`logger.flush()` tidak chain ke worker stream child → `flushLogs()` di test
tidak men-drain file. Solusi: `pino-roll` jalan di main thread, attach via
`pino.multistream([prettyStream, fileStream])`. `flushLogs()` extended:

```ts
const STREAMS_KEY = Symbol.for('@bot/utils/logger/streams');
// attach flushable streams to logger instance via Symbol property
// flushLogs:
//  1) logger.flush() → SonicBoom main-thread
//  2) iterate flushable streams, call .flushSync() best-effort
//  3) 50ms safety wait
```

Trade-off: file write blocking event loop ringan (overhead tulis WAL ~µs).
Untuk volume bot chat, gak ada masalah throughput. Reliability menang.

### WhatsApp adapter — `packages/adapters/src/wa/context.ts`

Group name cache 5 menit TTL dengan async lazy fetch:

```ts
const groupNameCache = new Map<string, { name: string; expiresAt: number }>();
const GROUP_NAME_TTL_MS = 5 * 60 * 1000;

async function fetchGroupName(socket: WASocket, chatId: string): Promise<string | undefined> {
  const cached = groupNameCache.get(chatId);
  if (cached && cached.expiresAt > Date.now()) return cached.name;
  try {
    const meta = await socket.groupMetadata(chatId);
    const name = meta?.subject || undefined;
    if (name) groupNameCache.set(chatId, { name, expiresAt: Date.now() + GROUP_NAME_TTL_MS });
    return name;
  } catch {
    return undefined;
  }
}

export function primeWaGroupName(chatId: string, name: string): void { ... }
export function clearWaGroupCache(): void { ... }
```

## Phase 4 — Daily rotation + 7-day retention + Asia/Jakarta

**Request user**: `data/log/<timestamp>.log`, daily rotate, max 7 hari, file
hari ke-8 (yang terlama) auto-hilang, pakai TZ Asia/Jakarta.

### Patch — `packages/utils/src/logger.ts`

```diff
  const fileStream = (await pinoRoll({
-   file: join(logDir, 'bot.log'),
-   frequency: 'daily',
-   size: '50m',
-   mkdir: true,
-   limit: { count: 14 },
+   file: join(logDir, 'bot'),
+   frequency: 'daily',
+   mkdir: true,
+   dateFormat: 'yyyy-MM-dd',
+   extension: '.log',
+   limit: { count: 7 },
  })) as FlushableStream & NodeJS.WritableStream;
```

Filename pattern:

```
data/log/bot.2026-05-27.1.log    ← hari ini
data/log/bot.2026-05-26.1.log    ← kemarin
...
data/log/bot.2026-05-21.1.log    ← 7 hari lalu (paling tua)
```

`.1` = sequence number `pino-roll` (mandatory di v4, gak bisa di-disable). Kalau
`size` di-set dan log gede, rotate dalam hari yang sama jadi `.2.log`, `.3.log`.
`size` di-drop biar bersih cuma daily.

Hari ke-8: `bot.2026-05-21` otomatis di-`fs.unlink()` oleh `pino-roll` saat
boundary midnight tercapai (limit 7).

### Rotation pakai TZ runtime

`pino-roll` rotate pakai `process.env.TZ`. `Asia/Jakarta` di env →
midnight = 00:00 WIB. Verified via real test: TZ=Asia/Jakarta + tulis log
04:02 WIB → file `bot.2026-05-27.1.log` ke-create (bukan UTC date).

### Cleanup file legacy

`data/log/bot.1.log` (153KB, log dev lama dari 23 May) di-`Remove-Item`. Format
lama (tanpa date stamp) gak akan ke-touch sama pino-roll baru karena pattern
beda.

## Phase 5 — TZ hardening (env-driven + validated + visible)

**Masalah**: `process.env.TZ = process.env.TZ ?? 'Asia/Jakarta'` di
`bootstrap.ts` raw, no validation. Typo `Asai/Jakarta` → Node silently fallback
UTC, log timestamp salah 7 jam, debug 3 hari.

3 lapis defense, semuanya di-implement.

### Lapis 1 — Validated zod schema (`packages/utils/src/config.ts`)

```ts
export function isValidTimezone(tz: string): boolean {
  try {
    new Intl.DateTimeFormat('en-US', { timeZone: tz });
    return true;
  } catch {
    return false;
  }
}

const tzSchema = z
  .string()
  .default('Asia/Jakarta')
  .refine(isValidTimezone, {
    message: 'Invalid IANA timezone. Examples: Asia/Jakarta, Asia/Singapore, UTC, America/New_York.',
  });

export const ConfigSchema = z.object({
  NODE_ENV: ...,
  TZ: tzSchema,
  ...
});
```

Field `TZ` baru ditambah di `AppConfig` contract
(`packages/contracts/src/app-context.ts:NODE_ENV: NodeEnv;` →
`+ TZ: string;`). Compile-time guard `_SchemaMatchesContract` masih jalan
verify schema = contract.

### Lapis 2 — Preflight di `apps/bot/src/bootstrap.ts`

Eksekusi sebelum import lain (timing kritis: banyak modul cache `Date`
behavior pas first import):

```ts
const FALLBACK_TZ = 'Asia/Jakarta';
const requestedTz = (process.env.TZ ?? FALLBACK_TZ).trim() || FALLBACK_TZ;
try {
  new Intl.DateTimeFormat('en-US', { timeZone: requestedTz });
  process.env.TZ = requestedTz;
} catch {
  process.stderr.write(
    `[bootstrap] Invalid TZ='${requestedTz}' (not an IANA name). Falling back to '${FALLBACK_TZ}'.\n`,
  );
  process.env.TZ = FALLBACK_TZ;
}
```

Dual safety: zod schema validate ulang nanti pas `loadConfig()` → kalau TZ
bypass preflight dengan cara aneh, schema tetap reject di boot.

## Catatan operasional VPS deploy

1. **TZ di Pterodactyl panel**: kalau egg `CUSTOM_ENVIRONMENT_VARIABLES`
   set `TZ=...`, panel-level menang dari `.env`. Cek `docs/deploy-pterodactyl.md`.
   Konsisten = `Asia/Jakarta` di kedua tempat.

2. **TZ di VPS region beda (US/SG)**: biarkan `TZ=Asia/Jakarta` di `.env`
   kalau lo mau log + scheduler ngikut WIB. Set TZ host system (`timedatectl`)
   bebas, gak ngaruh ke aplikasi karena `process.env.TZ` override.

3. **`docker run`**: `-e TZ=Asia/Jakarta` di command line, atau set di
   `docker-compose.yml` `environment:` block. Image base `node:20-alpine`
   default UTC, harus override.

4. **systemd service**: `Environment=TZ=Asia/Jakarta` di unit file kalau
   `.env` gak ke-load. Gak masalah kalau dobel-set, sama valuenya.

5. **Daily rotation midnight**: dengan `TZ=Asia/Jakarta`, file ganti tepat
   00:00 WIB. Pas hari ke-8, file ke-1 (`2026-05-21`) dihapus oleh limit
   `count: 7`. Manual cleanup tidak perlu.

6. **Backup**: `.backup` SQLite (lihat `README.md` ops checks) WAL-safe,
   bisa jalan saat bot running. Log file boleh `cp` aman karena append-only,
   gak ada lock.
